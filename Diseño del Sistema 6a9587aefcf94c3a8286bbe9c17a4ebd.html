Diseño del Sistema

VISIÓN:

Construir una plataforma de software en modalidad remota o virtual que permita a los estudiantes de la Universidad Católica de Oriente participar en proyectos de desarrollo de software conjuntos con sectores externos, satisfaciendo la demanda tanto interna como externa. Esta iniciativa beneficia a los estudiantes, validando su trabajo como trabajo de grado o práctica. Además, se involucra a los estudiantes que cursan cursos en la ruta de formación de Ingeniería de Software y/o Desarrollo de Software, programación y/o generación de código, para fortalecer la vinculación con el Desarrollo Académico Curricular.

La plataforma permitirá la operacionalización del laboratorio para la industria de desarrollo de software de la UCO, proporcionando servicios de consultoría en ingeniería de requisitos, aseguramiento de la calidad del software SQA y control de calidad de software SQC. Se busca construir las arquitecturas correspondientes tanto de procesos como de software para la prestación de servicios de consultoría. La integración de nuevos enfoques metodológicos, marcos de referencia y estándares internacionales permitirá la integración de nuevos enfoques al currículo. En resumen, el objetivo es construir la arquitectura de soluciones de software y sus procesos para la prestación de servicios de consultoría en el laboratorio para la industria de desarrollo de software de la Facultad de Ingenierías de la UCO.

ELEVATOR PITCH:

Un "elevator pitch" (discurso de ascensor) es una presentación breve y concisa diseñada para ser entregada en el tiempo que tomaría un viaje en ascensor, es decir, en unos 30 segundos a 2 minutos. El objetivo principal es captar rápidamente la atención de la audiencia y comunicar de manera efectiva quién eres, qué haces y por qué eres relevante o valioso.

Un buen "elevator pitch" debe responder a preguntas clave como:

  1. ¿Quién eres?
    • Breve introducción sobre tu identidad.
  1. ¿Qué haces?
    • Descripción concisa de tu trabajo, proyecto o negocio.
  1. ¿Cuál es tu propuesta de valor?
    • Indicar por qué lo que haces es relevante o beneficioso.
  1. ¿Cuál es tu objetivo?
    • Declarar lo que estás buscando o lo que esperas lograr.

MAPA DE IMPACTO:

Un mapa de impacto se refiere a una representación visual de cómo un cambio afectará a diferentes partes del sistema o a diferentes áreas de la organización. Esto se utiliza comúnmente en la gestión del cambio para entender quiénes serán afectados por un cambio, de qué manera y en qué medida. Puede incluir detalles sobre los equipos, departamentos o procesos que experimentarán cambios significativos.

Independientemente de la interpretación específica, un mapa de impacto suele ser una herramienta visual valiosa para comprender y comunicar de manera efectiva los efectos de ciertas acciones, ya sea en el medio ambiente o en una organización.

MODELADO DE DOMINIO:

El modelado de dominio es un proceso esencial en el desarrollo de software que implica la creación de un modelo conceptual de los diferentes componentes y entidades clave dentro de un sistema. Este modelo representa las relaciones y las reglas de negocio que gobiernan el comportamiento del sistema.

EVENT STORMING

Es una técnica de colaboración visual utilizada en el desarrollo de software para explorar y modelar procesos de negocio complejos. Fue introducida por Alberto Brandolini y se ha vuelto popular en la comunidad ágil y de desarrollo de software.

El "Event Storming" es especialmente útil al principio de un proyecto, cuando se busca comprender la complejidad del dominio del negocio y fomentar la colaboración entre los miembros del equipo. Al proporcionar una representación visual y colaborativa, ayuda a alinear a los participantes y proporciona una base sólida para el diseño e implementación subsiguientes.

MODELO ENTIDAD RELACIÓN:

El modelo entidad-relación (MER o ER, por sus siglas en inglés) es un modelo de datos que representa las entidades, relaciones, atributos y restricciones en un sistema. Este modelo es especialmente útil para diseñar bases de datos y entender la estructura de la información en un contexto determinado.

  1. Entidad: Representa un objeto del mundo real o una abstracción del mundo real que tiene datos almacenados en una base de datos.
  1. Atributo: Características o propiedades de una entidad.
  1. Relación: Asociación entre dos o más entidades.
  1. Cardinalidad: Describe la cantidad de instancias de una entidad que pueden estar relacionadas con una instancia de la entidad opuesta.
  1. Clave Primaria: Atributo o conjunto de atributos que identifican de manera única una instancia de una entidad.
  1. Clave Foránea: Atributo(s) que establece(n) una conexión entre dos tablas en una base de datos relacional. Generalmente, es la clave primaria de otra tabla y se utiliza para establecer relaciones.

El modelo entidad-relación es una herramienta poderosa para diseñadores de bases de datos y desarrolladores de sistemas que desean representar y comprender la estructura de los datos en una aplicación. Permite visualizar las relaciones entre diferentes partes de un sistema y proporciona una base para la creación de esquemas de bases de datos relacionales.

DIAGRAMA DE COMPONENTES

Un diagrama de componentes es un tipo de diagrama de la UML (Unified Modeling Language) que representa visualmente la estructura y las dependencias de los componentes de un sistema. Los componentes son elementos físicos o lógicos que encapsulan un conjunto de clases y proporcionan una interfaz clara para interactuar con ellas.

Los siguientes elementos se encuentran dentro de diagrama de componentes:

  1. Componente: Representa un módulo o parte del sistema que encapsula un conjunto de clases y ofrece una interfaz bien definida.
  1. Interfaz: Define cómo otros componentes pueden interactuar con el componente. Puede incluir operaciones y servicios proporcionados o requeridos.
  1. Dependencia: Una línea con una flecha que indica que un componente depende de otro. Por ejemplo, si un componente utiliza clases de otro componente, hay una dependencia entre ellos.
  1. Asociación: Representa la relación entre dos componentes. Puede indicar que los componentes trabajan juntos o se comunican de alguna manera.
  1. Uso: Indica que un componente utiliza otro componente para proporcionar una funcionalidad específica. Se representa mediante una línea punteada con una flecha.
  1. Herencia de Componente: Indica que un componente hereda de otro. Esta relación es similar a la herencia de clases, pero se aplica a nivel de componentes.
  1. Artefactos: Elementos físicos que pueden incluir archivos, bibliotecas, ejecutables, etc. Pueden estar asociados con componentes.

DISEÑO DE ALTO NIVEL / ARQUITECTONICO

Es una fase crítica en el desarrollo de software donde se definen las estructuras y componentes claves de un sistema. Este nivel de diseño se centra en aspectos globales y en la toma de decisiones que influirán en la arquitectura general del sistema. El diseño de alto nivel sienta las bases para la implementación y desarrollo del producto. Un buen diseño arquitectónico facilita la comprensión del sistema, permite la colaboración eficiente entre miembros del equipo y proporciona una guía clara para la implementación detallada del software.
